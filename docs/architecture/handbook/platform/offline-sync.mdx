### L2 採用の定量基準

以下の**すべて**を満たす操作にのみ L2 を適用する。
「UX が重要だから」という定性判断での採用は禁止。

| 基準 | 閾値 |
|---|---|
| ユーザーの操作頻度 | ≥ 10 回 / 日（アクティブユーザー中央値） |
| オフライン環境での発生可能性 | ≥ 5%（モバイルネットワーク切断を含む） |
| サーバー応答待ちのUX影響 | 体感遅延 ≥ 500ms かつユーザーが結果を即座に確認する操作 |
| コンフリクト解決方針の合意 | 事前に決定済みであること（後述） |

🔴 **MUST NOT**: L3（expo-sqlite）を UX 要件の確認なしに採用しない（実装コストが大きい）

## 2. AsyncStorage キー管理

`rq-cache`・バックグラウンド同期データなど、複数箇所に散在するキーを一元管理する。
ハードコードは禁止。

```ts
// src/lib/storageKeys.ts
export const StorageKeys = {
  queryCache:    'rq-cache',
  bgSync: {
    orders:    'bg-sync:orders',
    // 追加エンティティはここに列挙する
  },
} as const;
```

### AsyncStorage サイズ上限への注意

iOS の AsyncStorage 上限は **約 6 MB**。
`rq-cache` に大量データを持つクエリが混入すると容易に超過する。

🔴 **MUST**: `persistQueryClient` の `dehydrateOptions` で永続化対象クエリを明示的に絞る

```ts
// src/lib/queryPersister.ts
export function setupQueryPersistence() {
  persistQueryClient({
    queryClient,
    persister: asyncStoragePersister,
    maxAge:    1000 * 60 * 60 * 24,
    buster:    buildCacheBuster(),
    dehydrateOptions: {
      // 永続化が不要・またはサイズが大きいクエリを除外する
      shouldDehydrateQuery: (query) => {
        const key = query.queryKey[0] as string;
        return !EXCLUDE_FROM_PERSISTENCE.includes(key);
      },
    },
  });
}

// 永続化から除外するクエリキーのプレフィックス
const EXCLUDE_FROM_PERSISTENCE = [
  'fileDownload',
  'largePaginatedList',
];
```

### buster のバージョン戦略

アプリバージョン全体を `buster` に使うと、パッチリリースでもキャッシュ全破棄が発生する。
**メジャー.マイナーのみ**を使い、パッチ更新ではキャッシュを維持する。

```ts
// src/lib/queryPersister.ts
function buildCacheBuster(): string {
  const version = process.env.EXPO_PUBLIC_APP_VERSION ?? '0.0.0';
  // "1.3.2" → "1.3" のみ使う（パッチ変更でキャッシュ破棄しない）
  const [major, minor] = version.split('.');
  return `${major}.${minor}`;
}
```

## 4. ネットワーク状態の監視

```ts
// src/hooks/useNetworkStatus.ts
import NetInfo, { NetInfoState } from '@react-native-community/netinfo';
import { useEffect, useState }   from 'react';
import { queryClient }           from '@/lib/queryClient';

export type NetworkStatus = {
  isOnline:    boolean;
  type:        string | null;
  isExpensive: boolean;
};

export function useNetworkStatus(): NetworkStatus {
  const [status, setStatus] = useState<NetworkStatus>({
    isOnline:    true,
    type:        null,
    isExpensive: false,
  });

  useEffect(() => {
    return NetInfo.addEventListener((state: NetInfoState) => {
      const isOnline = state.isConnected ?? false;
      setStatus({
        isOnline,
        type:        state.type,
        isExpensive: state.details?.isConnectionExpensive ?? false,
      });

      if (isOnline) {
        queryClient.resumePausedMutations();
        queryClient.invalidateQueries({
          predicate: (query) => query.isStale(),
        });
      }
    });
  }, []);

  return status;
}
```

```tsx
// src/components/providers/NetworkStatusProvider.tsx
export function NetworkStatusProvider({ children }: { children: React.ReactNode }) {
  const { isOnline } = useNetworkStatus();
  const { t }        = useTranslation();

  return (
    <>
      {children}
      {!isOnline && (
        <Banner
          type="warning"
          message={t('common.offline')}
        />
      )}
    </>
  );
}
```

🟡 **SHOULD**: 再接続時に `resumePausedMutations()` を呼び、オフライン中の Mutation を再実行する

## 5. L2: 楽観的更新（Optimistic Update）

### コンフリクト解決方針（必須・事前合意）

L2 採用前に以下のいずれかの方針を決定し、ADR に記録する。
方針未決定のまま実装してはならない。

| 方針 | 概要 | 適用例 |
|---|---|---|
| **Last-Write-Wins** | サーバーの最新値で常に上書き | お気に入りトグル、既読フラグ |
| **Client-Wins** | オフライン中の変更を優先 | ドラフト保存 |
| **Server-Wins** | サーバー値を優先し、ローカル変更を破棄 | 在庫数など外部変動が大きいデータ |
| **Merge** | フィールド単位で合成（複雑） | 複数フィールドを独立して編集する場合 |

🔴 **MUST**: `onSettled` での `invalidateQueries` により、楽観的更新の値は最終的に必ずサーバー値で上書きされる。これは Last-Write-Wins の実装である。他の方針を採用する場合は追加実装が必要。

### 実装パターン

```ts
// features/order/hooks/useToggleFavorite.ts
export function useToggleFavorite() {
  return useMutation({
    mutationFn: ({ orderId, isFav }: { orderId: string; isFav: boolean }) =>
      httpClient.patch(`/orders/${orderId}/favorite`, { isFav }, favoriteResponseSchema),

    onMutate: async ({ orderId, isFav }) => {
      // 進行中の再フェッチをキャンセルして競合を防ぐ
      await queryClient.cancelQueries({ queryKey: queryKeys.orders.detail(orderId) });

      // ロールバック用スナップショットを保存
      const snapshot = queryClient.getQueryData<Order>(queryKeys.orders.detail(orderId));

      // キャッシュを楽観的に更新
      queryClient.setQueryData<Order>(queryKeys.orders.detail(orderId), (old) =>
        old ? { ...old, isFavorite: isFav } : old,
      );

      return { snapshot };
    },

    onError: (_err, { orderId }, context) => {
      // スナップショットへロールバック
      if (context?.snapshot) {
        queryClient.setQueryData(queryKeys.orders.detail(orderId), context.snapshot);
      }
      notify.error('お気に入りの更新に失敗しました');
    },

    onSettled: (_data, _err, { orderId }) => {
      // 成功・失敗問わずサーバーと同期（Last-Write-Wins）
      queryClient.invalidateQueries({ queryKey: queryKeys.orders.detail(orderId) });
    },
  });
}
```

🔴 **MUST**: `onMutate` でスナップショットを保存し、`onError` でロールバックする  
🔴 **MUST**: `cancelQueries` で楽観的更新と進行中の再フェッチの競合を防ぐ  
🟡 **SHOULD**: `onSettled` でサーバーとの最終同期を行い、楽観的更新の誤差を修正する

## 6. Mutation のオフラインキューイング

`networkMode: 'offlineFirst'` により、React Query はオフライン時の Mutation を
自動でキューに積み、再接続時に再実行する。
これは L2 とは独立した機能である（楽観的更新を伴わない Mutation にも適用される）。

```ts
useMutation({
  mutationFn:  submitOrder,
  networkMode: 'offlineFirst',  // queryClient のデフォルトから継承

  onMutate: async (variables) => {
    // L2 が必要な場合のみ楽観的更新を実装する
  },

  onError: (error) => {
    if (error instanceof NetworkError) {
      // オフラインによる失敗はユーザーに通知しない（再接続時に自動リトライ）
      return;
    }
    notify.error('送信に失敗しました');
  },
});
```

```tsx
// 再接続時の自動再実行
export function useResumeOnReconnect() {
  const { isOnline } = useNetworkStatus();

  useEffect(() => {
    if (isOnline) {
      queryClient.resumePausedMutations().then(() => {
        queryClient.invalidateQueries();
      });
    }
  }, [isOnline]);
}
```

## 7. バックグラウンドフェッチ（expo-background-fetch）

```bash
expo install expo-background-fetch expo-task-manager
```

```ts
// src/lib/backgroundSync.ts
import * as BackgroundFetch from 'expo-background-fetch';
import * as TaskManager     from 'expo-task-manager';
import * as SecureStore     from 'expo-secure-store';
import { StorageKeys }      from '@/lib/storageKeys';

const BACKGROUND_SYNC_TASK = 'background-sync';
const SYNC_INTERVAL_MIN    = 15;

// ─── 認証トークン取得 ──────────────────────────────────────────────────────
// バックグラウンドタスクは React Context にアクセスできないため、
// SecureStore から直接取得する（AuthContext 経由は不可）
async function getAuthToken(): Promise<string | null> {
  return SecureStore.getItemAsync('auth-token');
}

// ─── タスク定義 ───────────────────────────────────────────────────────────
TaskManager.defineTask(BACKGROUND_SYNC_TASK, async () => {
  try {
    const token = await getAuthToken();
    if (!token) return BackgroundFetch.BackgroundFetchResult.NoData;

    const orders = await httpClient.get('/orders/recent', ordersSchema);

    // AsyncStorage に保存（React Query キャッシュに直接書けないため）
    await AsyncStorage.setItem(
      StorageKeys.bgSync.orders,
      JSON.stringify({ data: orders, updatedAt: Date.now() }),
    );

    return BackgroundFetch.BackgroundFetchResult.NewData;
  } catch {
    return BackgroundFetch.BackgroundFetchResult.Failed;
  }
});

// ─── 登録・解除 ─────────────────────────────────────────────────────────
export async function registerBackgroundSync() {
  const status = await BackgroundFetch.getStatusAsync();

  if (
    status === BackgroundFetch.BackgroundFetchStatus.Restricted ||
    status === BackgroundFetch.BackgroundFetchStatus.Denied
  ) {
    logger.warn('Background fetch is not available');
    return;
  }

  await BackgroundFetch.registerTaskAsync(BACKGROUND_SYNC_TASK, {
    minimumInterval: SYNC_INTERVAL_MIN * 60,
    stopOnTerminate: false,
    startOnBoot:     false,
  });
}

export async function unregisterBackgroundSync() {
  const isRegistered = await TaskManager.isTaskRegisteredAsync(BACKGROUND_SYNC_TASK);
  if (isRegistered) {
    await BackgroundFetch.unregisterTaskAsync(BACKGROUND_SYNC_TASK);
  }
}
```

### フォアグラウンド復帰時のキャッシュ反映

バックグラウンドフェッチの最小間隔は 15 分であるため、
「5 分以内のデータのみ反映」という判定は事実上機能しない。
代わりに **タイムスタンプの差分** を表示し、古いデータかどうかをユーザーに委ねる。

```tsx
// app/_layout.tsx
import { AppState } from 'react-native';

export default function RootLayout() {
  useEffect(() => {
    const sub = AppState.addEventListener('change', async (nextState) => {
      if (nextState === 'active') {
        const raw = await AsyncStorage.getItem(StorageKeys.bgSync.orders);
        if (raw) {
          const { data, updatedAt } = JSON.parse(raw);
          // バックグラウンド同期データを React Query キャッシュに反映
          // 鮮度の判定はしない（最小間隔 15 分のため 5 分判定は無意味）
          queryClient.setQueryData(queryKeys.orders.all(), data);
          logger.info('bg-sync applied', { updatedAt, ageMs: Date.now() - updatedAt });
          await AsyncStorage.removeItem(StorageKeys.bgSync.orders);
        }

        queryClient.invalidateQueries({ refetchType: 'active' });
      }
    });
    return () => sub.remove();
  }, []);
}
```

🟡 **SHOULD**: バックグラウンドフェッチは認証が必要なデータのみ登録し、不要な場合は登録しない  
🔴 **MUST**: バックグラウンドタスク内での UI 操作（`notify` / `router.push`）は禁止  
🔴 **MUST**: 認証トークンは `SecureStore` から取得する。`AsyncStorage` への平文保存は禁止

## 8. expo-sqlite によるローカル DB（L3・要件確認後のみ）

### スキーマとマイグレーション

`db.execSync` による CREATE TABLE だけでは、アプリ更新時のカラム追加に対応できない。
バージョン管理された順次マイグレーションを実装する。

```ts
// src/lib/localDb.ts
import * as SQLite from 'expo-sqlite';

const db = SQLite.openDatabaseSync('app.db');

// ─── マイグレーション定義 ─────────────────────────────────────────────────
// 追加のみ可能。既存マイグレーションの変更・削除は禁止。
const MIGRATIONS: { version: number; sql: string }[] = [
  {
    version: 1,
    sql: `
      CREATE TABLE IF NOT EXISTS orders (
        id        TEXT PRIMARY KEY,
        data      TEXT NOT NULL,
        synced_at INTEGER,
        dirty     INTEGER DEFAULT 0
      );
      CREATE TABLE IF NOT EXISTS sync_queue (
        id         INTEGER PRIMARY KEY AUTOINCREMENT,
        operation  TEXT NOT NULL,
        table_name TEXT NOT NULL,
        record_id  TEXT NOT NULL,
        payload    TEXT,
        created_at INTEGER NOT NULL
      );
    `,
  },
  // バージョン 2 以降はここに追加する
  // { version: 2, sql: 'ALTER TABLE orders ADD COLUMN priority INTEGER DEFAULT 0;' },
];

export function initializeLocalDb() {
  // スキーマバージョン管理テーブル
  db.execSync(`
    CREATE TABLE IF NOT EXISTS schema_version (version INTEGER NOT NULL);
    INSERT INTO schema_version (version) SELECT 0 WHERE NOT EXISTS (SELECT 1 FROM schema_version);
  `);

  const currentVersion = (
    db.getFirstSync<{ version: number }>('SELECT version FROM schema_version')
  )?.version ?? 0;

  for (const migration of MIGRATIONS) {
    if (migration.version > currentVersion) {
      db.execSync(migration.sql);
      db.runSync('UPDATE schema_version SET version = ?', migration.version);
      logger.info('DB migration applied', { version: migration.version });
    }
  }
}
```

### CRUD

```ts
// ─── CRUD ─────────────────────────────────────────────────────────────────
export function upsertOrder(order: Order) {
  db.runSync(
    'INSERT OR REPLACE INTO orders (id, data, synced_at, dirty) VALUES (?, ?, ?, ?)',
    order.id, JSON.stringify(order), Date.now(), 0,
  );
}

export function getOrders(): Order[] {
  const rows = db.getAllSync<{ data: string }>('SELECT data FROM orders ORDER BY synced_at DESC');
  return rows.map((r) => JSON.parse(r.data));
}

// ─── 同期キュー（最大 500 件制限） ───────────────────────────────────────
const SYNC_QUEUE_MAX = 500;

export function enqueueSync(
  operation: 'CREATE' | 'UPDATE' | 'DELETE',
  tableName: string,
  recordId:  string,
  payload?:  unknown,
) {
  const count = db.getFirstSync<{ count: number }>('SELECT COUNT(*) as count FROM sync_queue')?.count ?? 0;
  if (count >= SYNC_QUEUE_MAX) {
    logger.warn('Sync queue is full. Dropping oldest item.', { count });
    db.runSync('DELETE FROM sync_queue WHERE id = (SELECT id FROM sync_queue ORDER BY created_at ASC LIMIT 1)');
  }

  db.runSync(
    'INSERT INTO sync_queue (operation, table_name, record_id, payload, created_at) VALUES (?, ?, ?, ?, ?)',
    operation, tableName, recordId, payload ? JSON.stringify(payload) : null, Date.now(),
  );
}

export function getPendingSyncItems() {
  return db.getAllSync<SyncQueueItem>(
    'SELECT * FROM sync_queue ORDER BY created_at ASC LIMIT 50',
  );
}

export function deleteSyncQueueItem(id: number) {
  db.runSync('DELETE FROM sync_queue WHERE id = ?', id);
}
```

### 同期処理とエラーハンドリング

HTTP エラーを一律に「キューから削除」せず、エラー種別に応じて処理を分ける。

```ts
// src/lib/syncEngine.ts

type SyncResult = { success: number; failed: number; skipped: number };

export async function syncPendingItems(): Promise<SyncResult> {
  const items = localDb.getPendingSyncItems();
  const result: SyncResult = { success: 0, failed: 0, skipped: 0 };

  if (items.length === 0) return result;

  for (const item of items) {
    try {
      switch (item.operation) {
        case 'CREATE':
          await httpClient.post(`/${item.table_name}`, JSON.parse(item.payload!), z.unknown());
          break;
        case 'UPDATE':
          await httpClient.patch(`/${item.table_name}/${item.record_id}`, JSON.parse(item.payload!), z.unknown());
          break;
        case 'DELETE':
          await httpClient.delete(`/${item.table_name}/${item.record_id}`, z.unknown());
          break;
      }
      localDb.deleteSyncQueueItem(item.id);
      result.success++;
    } catch (err) {
      if (err instanceof NetworkError) {
        // オフライン → 残りをスキップし、次回再試行
        result.skipped += items.length - result.success - result.failed - 1;
        break;
      }

      if (err instanceof HttpError) {
        if (err.status === 409) {
          // コンフリクト → 事前合意の解決方針に従う（例: Server-Wins）
          logger.warn('Sync conflict detected', { item, status: err.status });
          localDb.deleteSyncQueueItem(item.id);
          result.failed++;
          // ユーザーへのコンフリクト通知（UIスレッド復帰後に表示）
          syncEventEmitter.emit('conflict', { item });
          continue;
        }
        if (err.status >= 400 && err.status < 500) {
          // 4xx クライアントエラー → リトライ不可。キューから削除してログに残す
          logger.error('Sync failed with client error', { item, status: err.status });
          localDb.deleteSyncQueueItem(item.id);
          result.failed++;
          continue;
        }
        if (err.status >= 500) {
          // 5xx サーバーエラー → キューに残し次回リトライ
          logger.warn('Sync failed with server error, will retry', { item, status: err.status });
          result.skipped++;
          continue;
        }
      }

      logger.error('Sync failed with unknown error', { item, err });
      result.failed++;
    }
  }

  if (result.success > 0) {
    queryClient.invalidateQueries();
  }

  return result;
}
```

### 同期失敗のユーザー可視化

同期失敗はログに残すだけでなく、ユーザーに通知する。

```tsx
// src/components/SyncStatusBadge.tsx
// 同期キューに残件があることをバッジで表示する
export function SyncStatusBadge() {
  const pendingCount = usePendingSyncCount();

  if (pendingCount === 0) return null;

  return (
    <Badge
      type="warning"
      label={t('sync.pending', { count: pendingCount })}
      // 'オフライン中の変更が {{count}} 件未送信です'
    />
  );
}

// src/hooks/usePendingSyncCount.ts
export function usePendingSyncCount() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const update = () => setCount(localDb.getPendingSyncItems().length);
    update();
    const sub = syncEventEmitter.addListener('conflict', update);
    return () => sub.remove();
  }, []);

  return count;
}
```

🔴 **MUST**: L3 を採用する場合は同期コンフリクトの解決方針（ADR）を事前に決める  
🔴 **MUST**: 4xx と 5xx を同列に扱わない。4xx はリトライ不可、5xx はリトライ対象  
🟡 **SHOULD**: 同期キューは最大 500 件で上限を設ける  
🟡 **SHOULD**: 同期失敗件数をユーザーに可視化する（サイレント消失を防ぐ）

## 10. よくある実装ミスと対策

<table>
  <thead>
    <tr>
      <th>ミス</th>
      <th>影響</th>
      <th>対策</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td><code>staleTime: 0</code>（デフォルト）のまま</td>
      <td>画面フォーカスのたびに全 API を再フェッチ</td>
      <td>データ特性に合わせて設定（最低 60 秒）</td>
    </tr>
    <tr>
      <td>楽観的更新で <code>cancelQueries</code> を省略</td>
      <td>進行中の再フェッチが楽観的更新を上書きする</td>
      <td><code>onMutate</code> の冒頭で必ず <code>cancelQueries</code></td>
    </tr>
    <tr>
      <td>ロールバック未実装</td>
      <td>エラー時に画面がおかしい状態のまま固まる</td>
      <td><code>onError</code> でスナップショットを復元</td>
    </tr>
    <tr>
      <td>バックグラウンドタスクで <code>notify</code> を呼ぶ</td>
      <td>バックグラウンドでは UI 操作が不可。エラーになる</td>
      <td>タスク内では AsyncStorage への書き込みのみ</td>
    </tr>
    <tr>
      <td><code>buster</code> にパッチバージョンを含める</td>
      <td>バグ修正リリースのたびにキャッシュ全破棄が発生する</td>
      <td>メジャー.マイナーのみを <code>buster</code> に使う</td>
    </tr>
    <tr>
      <td>AsyncStorage のサイズ上限を無視して全クエリを永続化</td>
      <td>iOS 6MB 上限超過によりキャッシュ書き込みが失敗する</td>
      <td><code>shouldDehydrateQuery</code> で大サイズクエリを除外する</td>
    </tr>
    <tr>
      <td>syncEngine で 4xx と 5xx を同列に「削除して次へ」</td>
      <td>5xx の場合にリトライすべきデータが消失する</td>
      <td>4xx は削除、5xx はキューに残してリトライ、409 は別途ハンドリング</td>
    </tr>
    <tr>
      <td>L2 採用時にコンフリクト解決方針を未定義のまま実装</td>
      <td>複数デバイス編集時にデータが不整合になる</td>
      <td>ADR に方針を記録してから実装を開始する</td>
    </tr>
    <tr>
      <td>バックグラウンド同期データの復帰判定を「5 分以内」にする</td>
      <td>最小間隔 15 分のため条件が事実上成立せずデータが反映されない</td>
      <td>鮮度判定を撤廃し、取得データは常にキャッシュへ反映する</td>
    </tr>
    <tr>
      <td>認証トークンを AsyncStorage に平文保存してバックグラウンドで使う</td>
      <td>セキュリティリスク（ルート済み端末での漏洩）</td>
      <td><code>expo-secure-store</code>（Keychain/Keystore）から取得する</td>
    </tr>
  </tbody>
</table>
