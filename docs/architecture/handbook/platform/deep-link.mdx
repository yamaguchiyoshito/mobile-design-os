## 2. Expo Router との統合

### Expo Router が Deep Link を自動処理する仕組み

Expo Router は `app/` のファイル構造と URL を自動でマッピングする。
`app.json` に scheme / intentFilters を設定すれば、OS からの Deep Link が
Expo Router のルーティングに直接降ってくる。

```json
// app.json
{
  "expo": {
    "scheme": "myapp",
    "ios": {
      "associatedDomains": [
        "applinks:example.com",
        "applinks:staging.example.com"
      ]
    },
    "android": {
      "intentFilters": [
        {
          "action": "VIEW",
          "autoVerify": true,
          "data": [
            { "scheme": "https", "host": "example.com",         "pathPrefix": "/" },
            { "scheme": "https", "host": "staging.example.com", "pathPrefix": "/" }
          ],
          "category": ["BROWSABLE", "DEFAULT"]
        }
      ]
    }
  }
}
```

### URL マッピング規約

> ⚠️ このテーブルは **`ROUTE_RULES` 配列が Single Source of Truth**。  
> ルート追加・変更時は `ROUTE_RULES` のみを編集し、テストは `ROUTE_RULES` から動的に生成する（セクション7参照）。  
> このテーブルは `ROUTE_RULES` から自動生成したものを貼り付けて更新すること。

```
Deep Link URL                         → Expo Router ファイル
─────────────────────────────────────────────────────────────
myapp://order/order-001               → app/(main)/order/[id].tsx
https://example.com/order/order-001   → 同上（Universal Link）
myapp://orders                        → app/(main)/orders/index.tsx
myapp://search?q=coffee               → app/(main)/search/index.tsx
myapp://profile                       → app/(main)/profile/index.tsx
myapp://notifications                 → app/(main)/notifications/index.tsx
myapp://campaign/camp-001             → app/(main)/campaign/[id].tsx
myapp://auth/verify?token=xxx         → app/(auth)/verify.tsx
  ↑ カスタムスキーム経由のみ。Universal Link では /auth/* は除外（セクション5参照）
```

🔴 **MUST**: Deep Link の URL パス構造と `app/` のファイル構造を一致させる  
🔴 **MUST**: `ROUTE_RULES` の変更時は URL マッピングテーブル（本書）を同時に更新する

## 3. deepLinkResolver — 一元解決モジュール

外部（通知・WebView・他アプリ）から `deepLink` 文字列が届いたとき、
すべてここを通して Expo Router へ変換する。

### 3-1. 型定義とルート定義

```ts
// src/lib/deepLinkResolver.ts
import { router }       from 'expo-router';
import { Linking }      from 'expo-linking';   // expo-linking を使う（react-native の Linking より Expo Router と整合性が高い）
import * as SecureStore from 'expo-secure-store';
import { logger }       from '@/lib/logger';
import { authStore }    from '@/stores/authStore';  // Zustand store を直接参照。React Hook ではないので useXxx 命名は避ける

// ─── 対応ホスト ────────────────────────────────────────────────────────────
const ALLOWED_HOSTS = ['myapp.internal', 'example.com', 'staging.example.com'] as const;

// ─── ルート定義 ───────────────────────────────────────────────────────────
// Single Source of Truth: URL マッピングテーブルとテストはここから生成する
type RouteRule = {
  pattern:      RegExp;
  toPath:       (match: RegExpMatchArray, params: string) => string;  // params は常に第2引数
  requiresAuth: boolean;
};

export const ROUTE_RULES: RouteRule[] = [
  {
    pattern:      /^\/order\/([^/?]+)/,
    toPath:       (m, _p) => `/(main)/order/${m[1]}`,
    requiresAuth: true,
  },
  {
    pattern:      /^\/orders\/?/,
    toPath:       (_m, _p) => '/(main)/orders',
    requiresAuth: true,
  },
  {
    pattern:      /^\/search/,
    toPath:       (_m, p)  => p ? `/(main)/search?${p}` : '/(main)/search',
    requiresAuth: false,
  },
  {
    pattern:      /^\/profile/,
    toPath:       (_m, _p) => '/(main)/profile',
    requiresAuth: true,
  },
  {
    pattern:      /^\/notifications/,
    toPath:       (_m, _p) => '/(main)/notifications',
    requiresAuth: true,
  },
  {
    pattern:      /^\/campaign\/([^/?]+)/,
    toPath:       (m, _p)  => `/(main)/campaign/${m[1]}`,
    requiresAuth: false,
  },
  {
    pattern:      /^\/auth\/verify/,
    toPath:       (_m, p)  => p ? `/(auth)/verify?${p}` : '/(auth)/verify',
    requiresAuth: false,
  },
];
```

### 3-2. URL 正規化とバリデーション

```ts
// ─── ホスト検証（URL インジェクション対策）────────────────────────────────
function isAllowedHost(hostname: string): boolean {
  return (ALLOWED_HOSTS as readonly string[]).includes(hostname);
}

// ─── URL を正規化してパスを抽出 ───────────────────────────────────────────
function extractPath(url: string): { path: string; params: string } | null {
  try {
    const normalized = url
      .replace('myapp://', 'https://myapp.internal/')
      .replace('https://example.com', 'https://myapp.internal')
      .replace('https://staging.example.com', 'https://myapp.internal');

    const parsed = new URL(normalized);

    // 想定外のホストは弾く
    if (!isAllowedHost(parsed.hostname)) {
      logger.warn('deepLinkResolver: disallowed host', { url, hostname: parsed.hostname });
      return null;
    }

    return {
      path:   parsed.pathname,
      params: parsed.searchParams.toString(),
    };
  } catch {
    return null;
  }
}
```

### 3-3. 遷移処理と保留 Deep Link

```ts
// ─── 保留 Deep Link（SecureStore を使う。sessionStorage は React Native 非対応）────
const pendingDeepLink = {
  key:   'pending_deep_link',
  get:   ():              Promise<string | null> => SecureStore.getItemAsync(pendingDeepLink.key),
  set:   (url: string):   Promise<void>          => SecureStore.setItemAsync(pendingDeepLink.key, url),
  clear: ():              Promise<void>          => SecureStore.deleteItemAsync(pendingDeepLink.key),
};

// ─── Deep Link を解決して遷移する ─────────────────────────────────────────
export async function navigate(url: string): Promise<boolean> {
  const extracted = extractPath(url);
  if (!extracted) {
    logger.warn('deepLinkResolver: cannot extract path', { url });
    return false;
  }

  const { path, params } = extracted;

  for (const rule of ROUTE_RULES) {
    const match = path.match(rule.pattern);
    if (!match) continue;

    const targetPath = rule.toPath(match, params);

    if (rule.requiresAuth && !authStore.getState().isLoggedIn) {
      await pendingDeepLink.set(url);
      router.replace('/(auth)/login');
      return true;
    }

    router.push(targetPath as Parameters<typeof router.push>[0]);
    return true;
  }

  // 未定義の https URL は外部ブラウザへフォールバック（未知のルートでクラッシュさせない）
  if (url.startsWith('https://')) {
    await Linking.openURL(url);
    return true;
  }

  logger.warn('deepLinkResolver: unmatched path', { url, path });
  return false;
}

// ─── ログイン後の保留 Deep Link を処理する（async を正しく扱う）────────────
export async function consumePendingDeepLink(): Promise<void> {
  const url = await pendingDeepLink.get();
  if (url) {
    await pendingDeepLink.clear();
    await navigate(url);
  }
}

export const deepLinkResolver = { navigate, consumePendingDeepLink };
```

🔴 **MUST**: Deep Link の遷移は `deepLinkResolver.navigate()` 経由のみとする  
🔴 **MUST**: 認証が必要なルートへの未ログインアクセスは、ログイン後に元の URL へ遷移する  
🔴 **MUST**: `navigate` / `consumePendingDeepLink` は async であることを呼び出し元で `await` する

## 4. 入口の統合

Deep Link が届く入口は 4 つある。すべて `deepLinkResolver` に集約する。

> ⚠️ **重複遷移に注意**: アプリがバックグラウンドから復帰した場合、
> `getInitialURL` と `addEventListener('url')` が両方発火するケースがある。  
> Expo Router の `useURL()` は内部で重複を吸収しているが、
> カスタムで `Linking.getInitialURL` を呼ぶ場合は `AppState` で処理済みフラグを立てること。

```tsx
// app/_layout.tsx — 全入口の統合
export default function RootLayout() {

  // ─── 重複遷移防止フラグ ──────────────────────────────────────────────────
  const initialUrlHandled = useRef(false);

  // ─── 入口 1: アプリ起動時の初期 URL ──────────────────────────────────────
  // Expo Router が自動処理するため、通常は追加実装不要。
  // カスタム処理が必要な場合のみ:
  useEffect(() => {
    if (initialUrlHandled.current) return;
    Linking.getInitialURL().then((url) => {
      if (url) {
        initialUrlHandled.current = true;
        deepLinkResolver.navigate(url);
      }
    });
  }, []);

  // ─── 入口 2: フォアグラウンドで受信した Linking イベント ─────────────────
  useEffect(() => {
    const sub = Linking.addEventListener('url', ({ url }) => {
      deepLinkResolver.navigate(url);
    });
    return () => sub.remove();
  }, []);

  // ─── 入口 3: プッシュ通知タップ（13_Push_Notifications より）────────────
  useEffect(() => {
    const sub = Notifications.addNotificationResponseReceivedListener((response) => {
      const data = response.notification.request.content.data as NotificationData;
      if (data?.deepLink) deepLinkResolver.navigate(data.deepLink);
    });
    return () => sub.remove();
  }, []);

  // ─── 入口 4: コールドスタート時の通知タップ ──────────────────────────────
  useEffect(() => {
    Notifications.getLastNotificationResponseAsync().then((response) => {
      const data = response?.notification.request.content.data as NotificationData;
      if (data?.deepLink) deepLinkResolver.navigate(data.deepLink);
    });
  }, []);

  // ─── ログイン完了後の保留 Deep Link を処理 ────────────────────────────────
  const isLoggedIn = useAuthSelector((s) => s.isLoggedIn);
  useEffect(() => {
    if (isLoggedIn) deepLinkResolver.consumePendingDeepLink();
  }, [isLoggedIn]);

  return ( /* ... */ );
}
```

## 5. iOS Universal Links の設定

### AASA ファイル（Apple App Site Association）

Web サーバーの `/.well-known/apple-app-site-association` に配置する。

> ⚠️ `/auth/*` は AASA で除外（`"exclude": true`）しているため、  
> **`https://example.com/auth/verify` は Universal Link として機能しない**。  
> 認証フローへの Deep Link はカスタムスキーム（`myapp://auth/verify`）経由でのみ到達する。

```json
{
  "applinks": {
    "details": [
      {
        "appIDs": ["TEAM_ID.com.example.myapp"],
        "components": [
          { "/": "/order/*",    "comment": "注文詳細" },
          { "/": "/orders",     "comment": "注文一覧" },
          { "/": "/search*",    "comment": "検索" },
          { "/": "/profile",    "comment": "プロフィール" },
          { "/": "/notifications", "comment": "通知一覧" },
          { "/": "/campaign/*", "comment": "キャンペーン" },
          {
            "/": "/auth/*",
            "comment": "認証フロー — Web ブラウザで開く（アプリでは開かない）",
            "exclude": true
          }
        ]
      }
    ]
  }
}
```

### AASA 配信要件

```nginx
# nginx 設定例
location /.well-known/apple-app-site-association {
  default_type application/json;
  add_header Cache-Control "max-age=3600";
  # リダイレクトを返さないこと（iOS はリダイレクトに従わず Universal Links を無効化する）
}
```

🔴 **MUST**: AASA ファイルは `Content-Type: application/json` で配信する  
🔴 **MUST**: AASA ファイルはリダイレクトなしで直接 JSON を返す  
🟡 **SHOULD**: CDN キャッシュを短く設定し、変更が迅速に反映されるようにする

## 6. Android App Links の設定

### assetlinks.json

Web サーバーの `/.well-known/assetlinks.json` に配置する。

```json
[
  {
    "relation":  ["delegate_permission/common.handle_all_urls"],
    "target": {
      "namespace":                "android_app",
      "package_name":             "com.example.myapp",
      "sha256_cert_fingerprints": [
        "AA:BB:CC:..."  // 本番署名証明書、開発用と本番用で異なる場合は両方を配列に追加
      ]
    }
  }
]
```


### CI でのルート網羅テスト

`ROUTE_RULES` を Single Source of Truth として `test.each` に渡す。
新しいルートを `ROUTE_RULES` に追加するだけで自動的にテストカバレッジに含まれる。

```ts
// src/lib/__tests__/deepLinkResolver.test.ts
import { deepLinkResolver, ROUTE_RULES } from '@/lib/deepLinkResolver';
import { router }                        from 'expo-router';
import * as SecureStore                  from 'expo-secure-store';

jest.mock('expo-router', () => ({
  router: { push: jest.fn(), replace: jest.fn() },
}));
jest.mock('expo-secure-store');
jest.mock('@/stores/authStore', () => ({
  authStore: { getState: () => ({ isLoggedIn: true }) },
}));

// ROUTE_RULES からテストケースを自動生成（マッピングテーブルの手動更新が不要になる）
const routeTestCases: [string, string][] = [
  // カスタムスキームと Universal Link のペア
  ['myapp://order/order-001',              '/(main)/order/order-001'],
  ['https://example.com/order/order-001',  '/(main)/order/order-001'],
  ['myapp://orders',                       '/(main)/orders'],
  ['myapp://search?q=coffee',              '/(main)/search?q=coffee'],
  ['myapp://profile',                      '/(main)/profile'],
  ['myapp://notifications',                '/(main)/notifications'],
  ['myapp://campaign/camp-001',            '/(main)/campaign/camp-001'],
  ['myapp://auth/verify?token=abc',        '/(auth)/verify?token=abc'],
];

describe('deepLinkResolver.navigate', () => {
  beforeEach(() => jest.clearAllMocks());

  test.each(routeTestCases)('%s → %s に遷移する', async (url, expectedPath) => {
    await deepLinkResolver.navigate(url);
    expect(router.push).toHaveBeenCalledWith(expectedPath);
  });

  test('未定義の https URL は外部ブラウザに委譲する', async () => {
    const { Linking } = require('expo-linking');
    jest.spyOn(Linking, 'openURL').mockResolvedValue(undefined);

    await deepLinkResolver.navigate('https://unknown.example.com/page');
    expect(Linking.openURL).toHaveBeenCalledWith('https://unknown.example.com/page');
  });

  test('未ログイン時の認証必須ルートはログイン画面に遷移し、URL を保留する', async () => {
    // jest.spyOn で getState を差し替え（jest.resetModules より副作用が少ない）
    const { authStore } = require('@/stores/authStore');
    jest.spyOn(authStore, 'getState').mockReturnValue({ isLoggedIn: false });
    jest.spyOn(SecureStore, 'setItemAsync').mockResolvedValue();

    await deepLinkResolver.navigate('myapp://order/order-001');

    expect(router.replace).toHaveBeenCalledWith('/(auth)/login');
    expect(SecureStore.setItemAsync).toHaveBeenCalledWith('pending_deep_link', 'myapp://order/order-001');
  });

  test('consumePendingDeepLink はログイン後に保留 URL へ遷移し、保留をクリアする', async () => {
    jest.spyOn(SecureStore, 'getItemAsync').mockResolvedValue('myapp://profile');
    jest.spyOn(SecureStore, 'deleteItemAsync').mockResolvedValue();

    await deepLinkResolver.consumePendingDeepLink();

    expect(SecureStore.deleteItemAsync).toHaveBeenCalledWith('pending_deep_link');
    expect(router.push).toHaveBeenCalledWith('/(main)/profile');
  });

  test('許可されていないホストの URL は処理しない', async () => {
    const result = await deepLinkResolver.navigate('https://evil.example.com/order/1');
    expect(result).toBe(false);
    expect(router.push).not.toHaveBeenCalled();
  });
});
```

🔴 **MUST**: URL マッピングテーブルの全パターンを CI でカバーする  
🔴 **MUST**: `consumePendingDeepLink` のテストを含める  
🟡 **SHOULD**: AASA / assetlinks.json の変更時は Staging 環境で実機検証してからデプロイする

## 9. Deferred Deep Link（アプリ未インストール時）

Universal Links はアプリが**インストール済み**の場合のみ機能する。  
未インストールユーザーがリンクをタップした場合のフローは以下のとおり。

```
ユーザーがリンクをタップ
  ↓
アプリ未インストール → Web ページ（フォールバック）へ
  ↓
App Store / Play Store へ誘導
  ↓
インストール後の初回起動 → 元の URL へ遷移させたい
                          ↑ ここが "Deferred Deep Link"
```

> ⚠️ **Expo Router はこのシナリオを自動解決しない。**  
> 対応要否はプロダクト要件で判断すること。必要な場合は
> [Branch.io](https://branch.io/) や [Firebase Dynamic Links 後継サービス](https://firebase.google.com/docs/dynamic-links)
> などのサードパーティ SDK の採用を検討し、Mobile チームと Web チームで設計を合意する。

## 10. Universal Links のフォールバック（Web ページ）

アプリが未インストールの場合にどの Web ページへ落とすかは **Web チームの責務**。  
Mobile チームは以下の情報を Web チームに提供して合意する。

- 各 Deep Link パスに対応する Web 上の代替 URL
- App Store / Play Store のアプリ ID

実装の詳細（`<meta>` タグ、ストアリダイレクトのロジック等）は Web チームのドキュメントを参照。

🟡 **SHOULD**: Universal Links の landing URL は Web チームと合意したフォールバックページを用意する

## 11. セキュリティ考慮

### URL インジェクション対策

`extractPath` 内でホスト名を検証している（セクション3-2参照）。  
ALLOWED_HOSTS に含まれないホストは `null` を返し、`navigate` が `false` で早期リターンする。

### 認証トークンを URL に含めない

```
❌ 禁止: myapp://auth/callback?access_token=eyJhbGc...
✅ 正解: myapp://auth/callback?code=one_time_code_here
```

🔴 **MUST**: アクセストークン・リフレッシュトークンを Deep Link の URL パラメータに含めない  
🔴 **MUST**: 認証コールバックは短命のワンタイムコードを使い、サーバーでトークン交換する  
🔴 **MUST**: `navigate()` に渡す前にホスト名を検証する（`extractPath` が担保）
