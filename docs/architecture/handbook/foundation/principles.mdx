## 2. レイヤー責務と依存方向

```
┌─────────────────────────────────────────┐
│  app/ (Expo Router Screen)              │  ← ナビゲーション・URL 設計の一次ソース
├─────────────────────────────────────────┤
│  features/ (Feature Components)        │  ← 業務ロジック・状態結合
├─────────────────────────────────────────┤
│  components/ (Design System)           │  ← 再利用可能 UI 部品・トークン消費者
├─────────────────────────────────────────┤
│  hooks/ / stores/ / queries/           │  ← 状態・通信・副作用
├─────────────────────────────────────────┤
│  lib/httpClient  lib/factory  lib/i18n │  ← インフラ共通層
└─────────────────────────────────────────┘
         依存方向: 上 → 下のみ
```

🔴 **MUST**: 下位レイヤーから上位レイヤーへのインポート禁止  
🔴 **MUST**: `app/` 配下のファイルは UI 結合のみ担う（ロジックを書かない）

## 3. 絶対禁止事項（MUST NOT）

### UI 部品の外来持ち込み禁止

```tsx
// ❌ 禁止 — Web DOM 前提のライブラリ
import { Button } from 'shadcn/ui';
import * as Dialog from '@radix-ui/react-dialog';

// ✅ 正解 — 自前 Design System
import { Button } from '@/components/ui/Button';
import { Modal } from '@/components/ui/Modal';
```

> **理由**: shadcn/ui・Radix UI は DOM 依存であり React Native では動作しない

### 通信の直書き禁止

```tsx
// ❌ 禁止
const res = await fetch('/api/users');

// ✅ 正解
const res = await httpClient.get('/users');
```

### 通知の直呼び禁止

```tsx
// ❌ 禁止
Toast.show({ text: 'Error' });

// ✅ 正解
notify.error('保存に失敗しました');
```

### 非決定論的値の直使用禁止

```tsx
// ❌ 禁止
const id = Math.random().toString();
const now = new Date();

// ✅ 正解（注入された Provider 経由）
const id = idProvider.generate();
const now = clock.now();
```

### 端末 API の直呼び禁止

```tsx
// ❌ 禁止 — 画面層から Expo モジュール直呼び
import * as Location from 'expo-location';
const loc = await Location.getCurrentPositionAsync();

// ✅ 正解 — wrapper 経由
import { locationService } from '@/lib/location';
const loc = await locationService.getCurrent();
```

## 4. ナビゲーション方針

🔴 **MUST**: ルーティングの正とするのは `app/` のファイル構造のみ  
🔴 **MUST**: `react-navigation` を直接使わない（Expo Router が内包する）  
🟡 **SHOULD**: 深いネストは `(group)` ディレクトリで論理分割する

```
app/
  (auth)/
    login.tsx
    signup.tsx
  (main)/
    _layout.tsx        ← Tab Navigator
    index.tsx          ← ホーム
    search/
      index.tsx
      [id].tsx
  _layout.tsx          ← Root layout（Provider 注入点）
```

## 5. 型安全・契約整合方針

```
OpenAPI 仕様
    ↓ orval / openapi-zod-client
Zod スキーマ（自動生成）
    ↓
TypeScript 型（推論）   +   Factory（テストデータ生成）
    ↓                           ↓
httpClient レスポンス検証    MSW ハンドラ / Jest / Storybook
```

🔴 **MUST**: Zod スキーマは OpenAPI から自動生成し、手書き禁止  
🔴 **MUST**: API レスポンスは実行時に Zod で検証する（`parse` / `safeParse`）

## 6. 選定理由（主要技術）

### Expo を選ぶ理由

React Native 公式が「まず Framework を使う」を推奨しており、Expo はその中核候補である。
Dev Build・OTA・EAS Workflows による CI/CD 統合が他の選択肢と比較して最も成熟している。

### Vitest を使わない理由

Expo 公式は unit testing に `jest-expo` を案内しており、
Expo Router の testing も `jest-expo` + `@testing-library/react-native` 前提である。
React Native Testing Library は完全な native runtime を提供しないため、
native API や OS 依存挙動は simulator / 実機の E2E で担保する必要がある。

### next-intl を使わない理由

next-intl は Next.js（App Router）専用であり React Native では動作しない。
`expo-localization` + `i18next` / `react-i18next` に置換する。
