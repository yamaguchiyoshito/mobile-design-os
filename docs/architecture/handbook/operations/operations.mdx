## 1. CI/CD 全体フロー

```
Push / MR
    │
    ▼
GitLab CI（Static + Unit + Story tests）
    │ Pass
    ▼
EAS Build（Dev Build / Staging）
    │
    ▼
Maestro E2E（EAS Workflows）iOS + Android
    │ Pass
    ▼
EAS Submit（App Store / Google Play）または EAS Update（OTA）
```

## 2. GitLab CI 設定方針

```yaml
# .gitlab-ci.yml（抜粋）
stages:
  - lint
  - test
  - build
  - e2e

lint:
  stage: lint
  script:
    - pnpm typecheck
    - pnpm lint

unit-test:
  stage: test
  script:
    - pnpm test --ci --coverage

storybook-test:
  # Storybook 8.x + addon-vitest が前提。7.x 以前は test:storybook スクリプトが異なる
  stage: test
  script:
    - pnpm storybook:build:web
    - pnpm test:storybook   # addon-vitest（Storybook 8.x）

eas-build-staging:
  stage: build
  script:
    - eas build --platform all --profile staging --non-interactive
  only:
    - main

eas-e2e-android:
  stage: e2e
  script:
    - eas workflow:run e2e --platform android
  only:
    - main

eas-e2e-ios:
  stage: e2e
  script:
    - eas workflow:run e2e --platform ios
  only:
    - main
```

## 4. OTA（EAS Update）方針

### 使用可能なケース

🟢 **MAY**: JS バンドルのみ変更（バグ修正・文言変更・ロジック変更）  
🔴 **MUST NOT**: native コード変更（新 Expo module 追加・`app.json` 変更）を OTA で配信しない

### チャンネル戦略

```
main ブランチ       → production チャンネル（全ユーザー）
staging ブランチ    → staging チャンネル（内部テスター）
feature/* ブランチ  → development チャンネル（開発者のみ）
```

```bash
# OTA 配信コマンド例
eas update --branch production --message "Fix: 注文確認画面のクラッシュ修正"

# Staged rollout（段階的配信）
eas update --branch production --rollout-percentage 10
```

🟡 **SHOULD**: 本番 OTA は 10% → 50% → 100% の段階的 rollout を採用する  
🟡 **SHOULD**: OTA 配信後は Sentry / Crashlytics でクラッシュ率を 24h モニタリングする

## 5. バージョニング規約とリリースフロー

### バージョン体系

```
[major].[minor].[patch]  例: 2.3.1

major: 破壊的変更 / 大型機能追加
minor: 後方互換のある機能追加
patch: バグ修正・軽微な変更
```

### ビルド番号

```json
// app.json
{
  "expo": {
    "version": "2.3.1",
    "ios":     { "buildNumber": "231" },
    "android": { "versionCode": 231 }
  }
}
```

🔴 **MUST**: `version` と `buildNumber` / `versionCode` は必ずセットで更新する  
🔴 **MUST**: バージョン更新は `main` へのマージ前に PR 内で完結させる  
🟡 **SHOULD**: リリース時は Git Tag を打つ（`v2.3.1`）

### リリース手順（担当・タイミング）

```
1. [担当: 開発者] PR で app.json の version / buildNumber を更新してレビュー依頼
2. [担当: レビュアー] バージョン番号の妥当性を確認してマージ承認
3. [担当: CI/CD] main マージ後に GitLab CI が自動でテスト → EAS Build を実行
4. [担当: リリース担当者] EAS Build 成功後に `eas submit` を手動実行
5. [担当: リリース担当者] Git Tag を打つ（例: git tag v2.3.1 && git push origin v2.3.1）
6. [担当: リリース担当者] CHANGELOG.md を更新して main へ push
```

## 6. 環境変数管理

```
.env.development   ← ローカル開発用（Git 管理外）
.env.staging       ← staging 用（Git 管理外、EAS Secret から注入）
.env.production    ← 本番用（Git 管理外、EAS Secret から注入）
.env.example       ← 必須変数の一覧（Git 管理、値なし）
```

```bash
# EAS Secret への登録
eas secret:create --scope project --name API_BASE_URL --value "https://api.example.com"

# 既存シークレットの更新
eas secret:push --scope project --env-file .env.production

# シークレット一覧の確認
eas secret:list
```

### シークレットローテーション手順

```
1. 新しいシークレット値を発行（APIキー再生成など）
2. eas secret:push で EAS 側を更新
3. eas build --profile production で新しいシークレットを含む新ビルドを作成
4. 旧シークレットを無効化（APIキーの場合は発行元サービスで失効処理）
5. 旧ビルドが旧シークレットを使用していないことを監視ログで確認
```

**漏洩時の対応（緊急）**  
🔴 **MUST**: 漏洩を検知したら即座に発行元サービスでキーを失効させる。EAS 更新より失効処理を優先すること。

🔴 **MUST**: API キー・シークレットを Git にコミットしない  
🔴 **MUST**: `EXPO_PUBLIC_` プレフィックスの変数は JS バンドルに含まれることを意識する  
🟡 **SHOULD**: シークレットは EAS Secret で管理し、CI 環境変数に平文で書かない  
🟡 **SHOULD**: シークレットのローテーションは 90 日ごとに実施する

## 8. インシデント対応・ロールバック

### OTA ロールバック手順

```bash
# 1. 現在の更新履歴を確認
eas update:list --branch production

# 2. 問題のある更新を特定し、直前の安定版 ID を確認
eas update:view <UPDATE_ID>

# 3. 安定版へのロールバック（再配信）
eas update --branch production --message "Rollback: <問題の説明>" --republish --group <STABLE_GROUP_ID>
```

**判断基準と実行者**

<table>
  <thead>
    <tr>
      <th>状況</th>
      <th>判断基準</th>
      <th>実行者</th>
      <th>タイムリミット</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>クラッシュ率が前日比 2 倍超</td>
      <td>Sentry アラート受信後、原因特定より先にロールバックを優先</td>
      <td>オンコール担当者</td>
      <td>アラート受信から 30 分以内</td>
    </tr>
    <tr>
      <td>OTA 配信成功率 90% 未満</td>
      <td>配信から 2h 後に EAS ダッシュボードで確認</td>
      <td>リリース担当者</td>
      <td>配信から 4h 以内</td>
    </tr>
    <tr>
      <td>重大な UI / ビジネスロジックバグ</td>
      <td>ビジネスインパクト判断はリードエンジニアが承認</td>
      <td>リードエンジニア承認 → オンコール実行</td>
      <td>ビジネス判断に従う</td>
    </tr>
  </tbody>
</table>

### EAS Build 失敗時のフォールバック

🔴 **MUST NOT**: EAS Build が失敗した状態で `main` マージを続行しない

```bash
# EAS Build が失敗した場合の緊急ローカルビルド（iOS）
# 実行権限: コード署名証明書を保持するリリース担当者のみ
eas credentials  # 証明書をローカルへ取得
npx expo run:ios --configuration Release

# Android
npx expo run:android --variant release
```

🟡 **SHOULD**: EAS Build の連続失敗（3回以上）は Slack の `#release-alert` チャンネルへ自動通知する

## 9. Deprecation ポリシー

### API 変更

```ts
/**
 * @deprecated v3.0 で削除予定。useNewSearch() を使用してください。
 * @see useNewSearch
 */
export function useSearch() { ... }
```

### コンポーネント変更

```tsx
// ✅ 廃止コンポーネントの移行ガイドを Story に記載する
export const Deprecated: Story = {
  name: '⚠️ Deprecated（v4.0 で削除）',
  parameters: {
    docs: {
      description: {
        story: '`NewButton` に移行してください。Props の対応表は `Architecture/Handbook/02_UI_System` を参照。',
      },
    },
  },
};
```

🔴 **MUST**: Deprecation から削除まで **最低 1 マイナーバージョン（≒ 1 ヶ月）** の猶予を設ける  
　→ 複数チームが依存する共有コンポーネントの場合は **2 マイナーバージョン（≒ 2 ヶ月）** を目安とする  
🔴 **MUST**: 廃止予定の Symbol には `@deprecated` JSDoc を必ず付ける  
🟡 **SHOULD**: Deprecation 宣言時に Slack の `#dev-announce` チャンネルへ移行先と削除予定バージョンを告知する

## 10. バンドルサイズ管理

```bash
# Expo managed workflow では expo export を使用する（react-native bundle は bare workflow 用）
npx expo export --platform ios --dev false

# バンドルサイズの確認（生成された dist/ ディレクトリを参照）
ls -lh dist/_expo/static/js/ios/

# カスタムスクリプトで前回比較
node scripts/bundle-size-check.js --threshold 50kb
```

🟡 **SHOULD**: バンドルサイズ増加を CI ゲートで 50KB 増分以上の場合は警告する  
🟡 **SHOULD**: 大型ライブラリ追加時は `pnpm why` でサイズ増分を確認する
