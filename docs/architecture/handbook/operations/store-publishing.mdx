# ストア公開運用ガイド TIPS

> **ステータス**: `APPROVED`  
> **対象**: App Store（Apple）・Google Play（Google）への公開・審査・運用全般  
> **前提**: [06_Operations](./06_Operations) を読んでいること（EAS Build / EAS Submit / バージョニングの基礎を定義）  
> **本書の範囲**: 06 は「CI/CD からどう提出するか」を定義した。本書は「審査を通すためのノウハウ・リジェクト対策・ストア運用の継続的な実務」を定義する。06 との重複箇所は本書を正とする

## 0. リリース前チェックリスト（最初に確認）

> **提出前に必ずこのリストを完走すること。** 詳細は各セクションを参照。

```
コード品質
  □ 全テスト（Unit / Integration / E2E）が通っている
  □ TypeScript エラー・ESLint エラーがない
  □ expo-doctor の警告が 0 件
  □ Staging 環境で 24h 以上 QA し、Sentry 新規エラーがない

バージョン
  □ app.json の version・buildNumber・versionCode を更新した（Section 6）
  □ 同一 buildNumber / versionCode の二重提出ではないことを確認した
  □ リリースノートに更新内容が反映されている
  □ CHANGELOG.md（社内用）を更新した

ストアメタデータ（マーケティング承認済みであること）
  □ スクリーンショットが最新 UI を反映している
  □ 説明文に誤字・旧情報がない
  □ 権限説明文（iOS Info.plist）に具体的使用理由が書かれている
  □ App Privacy（iOS）/ Data Safety（Android）が正確に記入されている
  □ Privacy Manifest（PrivacyInfo.xcprivacy）が最新状態である（iOS）

iOS 固有
  □ TestFlight 内部テストで最低 1 日動作確認済み
  □ テストアカウントが App Store Connect に登録されている
  □ iOS Distribution 証明書の有効期限が 30 日以上ある
  □ Privacy Manifest の required reason API 申告に漏れがない

Android 固有
  □ targetSdkVersion が Google の現行要件を満たしている（Section 4-1 参照）
  □ Google Play 内部テストで動作確認済み
  □ Data Safety セクションが正確に記入されている
  □ ロールアウト割合を 1% から開始することを確認した

提出後
  □ 審査ステータスを App Store Connect / Play Console で確認した
  □ 審査通過後、Sentry / Crashlytics で 24h モニタリングを開始した
  □ クラッシュ率を確認しながら段階的にロールアウトを拡大する
```

## 1. 全体スケジュールの目安

```
コードフリーズ
    ↓ 1〜2 日
Staging ビルド + QA（最低 24h）
    ↓ 並行作業
    ├─ ストアメタデータ更新（マーケティング承認）
    └─ スクリーンショット撮影・更新
    ↓ 1〜2 日
Production ビルド（EAS Build）
    ↓ 数時間〜1 日
ストア提出（EAS Submit）
    ↓
審査待ち
  iOS App Store: 平均 1〜3 日（初回・機能追加時は 1 週間を見込む）
  Google Play:   平均 数時間〜3 日
  ※ リジェクト後の再審査: 初回と同等の期間を再度見込む
    ↓
段階的ロールアウト開始
  iOS:     全ユーザーに即公開 or 段階リリース（Phased Release・最大 7 日間）
  Android: 段階ロールアウト（1% → 10% → 50% → 100%）
    ↓
全体公開・監視（24h）
```

🟡 **SHOULD**: リリース目標日の **1 週間前** には提出する。審査遅延・リジェクトのバッファを確保する  
🔴 **MUST NOT**: 金曜日の夕方〜週末に提出しない（審査が遅延すると週明けまで対応できない）

## 2. EAS Submit の設定と証明書管理

### eas.json の提出設定

```json
// eas.json
{
  "submit": {
    "production": {
      "ios": {
        "appleId":         "developer@example.com",
        "ascAppId":        "1234567890",
        "appleTeamId":     "XXXXXXXXXX"
      },
      "android": {
        "serviceAccountKeyPath": "./google-play-service-account.json",
        "track":                 "internal",
        "releaseStatus":         "draft"
      }
    },
    "staging": {
      "ios": {
        "appleId":         "developer@example.com",
        "ascAppId":        "1234567890",
        "appleTeamId":     "XXXXXXXXXX"
      },
      "android": {
        "serviceAccountKeyPath": "./google-play-service-account.json",
        "track":                 "internal"
      }
    }
  }
}
```

### 提出コマンド

```bash
# 最新のビルドをストアに提出
eas submit --platform ios     --profile production
eas submit --platform android --profile production

# 特定のビルド ID を指定して提出
eas submit --platform ios --profile production --id <build-id>

# 両プラットフォームを同時に提出
eas submit --platform all --profile production
```

### Google Play サービスアカウントの設定

```bash
# google-play-service-account.json は Git 管理外
echo "google-play-service-account.json" >> .gitignore

# EAS Secret Files として管理
eas secret:create \
  --scope project \
  --name GOOGLE_PLAY_SERVICE_ACCOUNT \
  --type file \
  --value ./google-play-service-account.json
```

🔴 **MUST**: `google-play-service-account.json` を Git にコミットしない  
🟡 **SHOULD**: サービスアカウントの権限は「リリース管理者」のみに絞る

### Apple Developer 証明書の管理

```
Apple Developer Program（年間 $99 USD）
  ├─ 証明書（Certificate）
  │     ├─ iOS Distribution（有効期限: 1 年）
  │     └─ Apple Push Services（有効期限: 1 年）
  ├─ Provisioning Profile（有効期限: 1 年）
  │     ├─ App Store Distribution
  │     └─ Ad Hoc（TestFlight / 社内配布）
  └─ App ID（不変）
```

証明書の期限は CI で週次自動チェックする。実装は `scripts/ci/check-cert-expiry.js` を参照。  
30 日を切ったら GitLab Issue を自動起票し、担当者（Section 7 のエスカレーションパス参照）に通知する。

🔴 **MUST**: 証明書の有効期限が 30 日を切る前に更新する  
🟡 **SHOULD**: EAS に証明書管理を委譲し（`eas credentials --platform ios`）、手動管理をなくす

### Google Play の管理項目

```
Google Play Console（登録料: 一回 $25 USD）
  ├─ App Signing Key（不変・Google が管理）
  ├─ Upload Key（EAS Submit が使う key）
  │     └─ 紛失した場合: Google Play Console で再リセット申請
  └─ Service Account（EAS Submit の認証用）
        └─ 有効期限なし・半年ごとに権限を見直す
```

🟡 **SHOULD**: Upload Key のバックアップを EAS と社内の安全な場所（1Password 等）に保管する  
🔴 **MUST**: App Signing Key は Google が管理する設定（Play App Signing）を使い、自己管理しない

## 3. App Store（iOS）審査のポイント

### 3-1. 権限の使用説明文（Info.plist Usage Descriptions）

```json
// app.json / app.config.ts — 全使用権限に具体的な説明を書く
{
  "ios": {
    "infoPlist": {
      // ❌ リジェクトされる説明文（目的が不明確）
      "NSCameraUsageDescription": "カメラを使用します",

      // ✅ 通過する説明文（ユーザーへのメリットと具体的用途を明記）
      "NSCameraUsageDescription":
        "プロフィール写真の撮影と書類のスキャンに使用します。",
      "NSLocationWhenInUseUsageDescription":
        "近くの店舗を表示するために現在地を使用します。アプリ使用中のみ取得します。",
      "NSPhotoLibraryUsageDescription":
        "プロフィール写真の選択に使用します。",
      "NSPhotoLibraryAddUsageDescription":
        "加工した写真をカメラロールに保存するために使用します。",
      "NSFaceIDUsageDescription":
        "アプリへのサインインを素早く安全に行うために使用します。",
      "NSMicrophoneUsageDescription":
        "音声でのメッセージ送信に使用します。",
      "NSContactsUsageDescription":
        "友人への送金時に連絡先から選択するために使用します。"
    }
  }
}
```

### 3-2. Privacy Manifest（PrivacyInfo.xcprivacy）

Apple は 2024 年よりサードパーティ SDK を含む全アプリに Privacy Manifest の提出を必須化した。  
未申告の Required Reason API を使用しているとリジェクトされる。

```xml
<!-- ios/PrivacyInfo.xcprivacy -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>NSPrivacyTracking</key>
  <false/>
  <key>NSPrivacyTrackingDomains</key>
  <array/>
  <key>NSPrivacyCollectedDataTypes</key>
  <array>
    <!-- Firebase Analytics を使う場合の例 -->
    <dict>
      <key>NSPrivacyCollectedDataType</key>
      <string>NSPrivacyCollectedDataTypeDeviceID</string>
      <key>NSPrivacyCollectedDataTypeLinked</key>
      <false/>
      <key>NSPrivacyCollectedDataTypeTracking</key>
      <false/>
      <key>NSPrivacyCollectedDataTypePurposes</key>
      <array>
        <string>NSPrivacyCollectedDataTypePurposeAnalytics</string>
      </array>
    </dict>
  </array>
  <key>NSPrivacyAccessedAPITypes</key>
  <array>
    <!-- UserDefaults を使う場合 -->
    <dict>
      <key>NSPrivacyAccessedAPIType</key>
      <string>NSPrivacyAccessedAPICategoryUserDefaults</string>
      <key>NSPrivacyAccessedAPITypeReasons</key>
      <array>
        <string>CA92.1</string>
      </array>
    </dict>
  </array>
</dict>
</plist>
```

Required Reason API の全一覧は [Apple 公式ドキュメント](https://developer.apple.com/documentation/bundleresources/privacy_manifest_files/describing_use_of_required_reason_api) を参照すること。  
使用している SDK（Firebase・Sentry 等）の Privacy Manifest を `expo-modules-core` で集約する設定が必要な場合がある（Expo の公式ドキュメントを確認する）。

🔴 **MUST**: Required Reason API を使用している場合、その理由コードを申告する  
🟡 **SHOULD**: SDK 更新のたびに Privacy Manifest の申告内容を見直す

### 3-3. App Privacy（プライバシーラベル）

App Store Connect の「App Privacy」セクションは審査前に**必ず記入する**。

```
必ず回答が必要な質問:
  ├─ データを収集するか？ → Firebase Analytics を使うなら YES
  ├─ 収集するデータの種類:
  │     ├─ 使用状況データ（画面表示・ボタン押下）→ Analytics
  │     ├─ クラッシュデータ                    → Sentry / Crashlytics
  │     ├─ 識別子（デバイス ID・ユーザー ID）   → Analytics / Crashlytics
  │     └─ 購入（購入履歴）                     → 決済機能がある場合
  └─ サードパーティと共有するか？ → Firebase / Sentry があれば YES
```

| Data Type | Collection | Linked to User | Tracking |
|-----------|-----------|----------------|---------|
| Usage Data | ✅ | ❌ | ❌ |
| Crash Data | ✅ | ❌ | ❌ |
| Identifiers（Device ID）| ✅ | ❌ | ❌ |
| Purchases | ✅（決済ありの場合）| ✅ | ❌ |

### 3-4. 起動直後の権限リクエスト

```
❌ リジェクトされる → アプリ起動直後・スプラッシュ画面・ログイン前に権限を要求する
✅ 通過する         → 機能が必要になったタイミングで要求する（14_Device_Permissions 参照）
```

### 3-5. In-App Purchase（IAP）の適用漏れ

```
❌ リジェクト対象 → デジタルコンテンツ・サブスクリプションを Apple 経由以外で決済する
✅ 例外（審査通過）→ 実物商品・サービス・B2B 業務アプリの機能解放は IAP 不要
```

🟡 **SHOULD**: デジタルコンテンツが存在する場合は法務確認の上で IAP を実装する  
🔴 **MUST**: 「App Store 以外で購入してください」というリンクや文言をアプリ内に置かない（ガイドライン 3.1.1 違反）

### 3-6. テストアカウントの提供

```
App Store Connect → App Review Information → Demo Account
  ├─ Username: reviewer@example.com
  ├─ Password: Review1234!
  └─ Notes:
       - ゲストログインボタンでもログイン可能です
       - 決済フローは sandbox モードで動作します
       - 管理者権限の機能は「設定 > テストモード」から確認できます
```

🔴 **MUST**: すべての機能が確認できるテストアカウントを提供する  
🟡 **SHOULD**: テストアカウントには本番に近い状態のデータを用意する（空のアカウントは機能確認不能でリジェクトされる）

### 3-7. 暗号化輸出規制の申告

```json
// app.json — HTTPS のみ使用する場合（対象外申告）
{
  "ios": {
    "infoPlist": {
      "ITSAppUsesNonExemptEncryption": false
    }
  }
}
```

この設定がないと App Store Connect で「Missing Compliance」警告が出てビルドが提出待ちになる。  
独自暗号化を実装している場合は `true` にして輸出規制申告書を別途提出すること。

## 4. Google Play（Android）審査のポイント

### 4-1. ターゲット API レベルの最新化

Google Play は毎年 8 月頃に targetSdkVersion の要件を更新する。  
現行要件は [Google 公式ポリシーページ](https://developer.android.com/google/play/requirements/target-sdk) で必ず確認すること（本書に固定値は記載しない）。

```json
// app.json — 公式ページで確認した現行要件に更新する
{
  "android": {
    "targetSdkVersion": <公式ページで確認>,
    "minSdkVersion": 24
  }
}
```

🔴 **MUST**: 毎年 8 月頃の Google の要件更新に追従する（10_Library_Update の年次チェック項目に追加する）

### 4-2. 権限の宣言（AndroidManifest）

```json
// app.json — 使用する権限のみ宣言する（未使用権限は却下対象）
{
  "android": {
    "permissions": [
      "android.permission.CAMERA",
      "android.permission.READ_MEDIA_IMAGES",
      "android.permission.ACCESS_FINE_LOCATION",
      "android.permission.RECEIVE_BOOT_COMPLETED",
      "android.permission.VIBRATE"
    ]
  }
}
```

🔴 **MUST**: 宣言した権限を実際に使用する。未使用権限はリジェクト対象  
🟡 **SHOULD**: `expo-doctor` で不要な権限が自動追加されていないか定期確認する

### 4-3. データ安全セクション（Data Safety）

```
Google Play Console → アプリのコンテンツ → データ安全

収集するデータ:
  ├─ 位置情報: おおよその位置 → YES（位置情報機能がある場合）
  ├─ アプリのアクティビティ: アプリのインタラクション → YES（Analytics）
  ├─ クラッシュログ → YES（Sentry / Crashlytics）
  ├─ 端末 ID / その他の ID → YES（Analytics）
  └─ 個人情報: ユーザー ID → YES（ログイン機能がある場合）

データの取り扱い:
  ├─ ユーザーのデータを第三者と共有するか？ → YES（Firebase / Sentry）
  └─ データを暗号化して転送するか？         → YES（HTTPS 必須）
```

🔴 **MUST**: Data Safety セクションを正確に記入する。虚偽記入はアカウント停止対象

### 4-4. 段階的ロールアウトの活用

```
Google Play Console → リリース → 本番 → 新しいリリースを作成

ロールアウトの割合:
  1%  → 24h 監視（クラッシュ率・ANR 率を確認）
  10% → 24h 監視
  50% → 24h 監視
  100% → 全体公開
```

🟡 **SHOULD**: 初回リリースは 1% から開始し、クラッシュ率が閾値以内であることを確認しながら拡大する

### 4-5. 内部テスト → クローズドテスト → オープンテスト → 本番のトラック管理

```
[内部テスト] → [クローズドテスト] → [オープンテスト] → [本番]
      QA 1〜2 名    社内 + 外部 QA      大規模ベータ     全ユーザー
      即承認         即承認              即承認           審査あり
```

```bash
# EAS Submit でトラックを指定
eas submit --platform android --profile staging
# → eas.json の "track": "internal" → 内部テストトラックへ提出
```

## 5. ストアメタデータの管理

### ファイル構成

```
store/
  ios/
    metadata/
      ja/
        description.txt          ← アプリ説明文（4000 文字以内）
        keywords.txt             ← キーワード（100 文字以内、カンマ区切り）
        promotional_text.txt     ← プロモーションテキスト（170 文字以内）
        release_notes.txt        ← アップデート内容（4000 文字以内）
      en-US/
        （同構成）
    screenshots/
      iPhone_6.9/               ← 必須サイズ（Apple 公式ドキュメントで要件を確認）
      iPad_13/                  ← 必須サイズ（同上）
  android/
    metadata/
      ja-JP/
        full_description.txt    ← 詳細説明（4000 文字以内）
        short_description.txt   ← 短い説明（80 文字以内）
        title.txt               ← アプリ名（30 文字以内）
        changelogs/
          <versionCode>.txt     ← バージョンごとのリリースノート
      en-US/
        （同構成）
```

スクリーンショットの要件サイズは Apple / Google が随時更新するため、  
提出前に [App Store Connect ヘルプ](https://help.apple.com/app-store-connect/#/devd274dd925) および [Google Play ヘルプ](https://support.google.com/googleplay/android-developer/answer/9866151) で必ず確認すること。

🟡 **SHOULD**: メタデータは Git で管理し、マーケティング担当が PR レビューする（承認なしに提出しない）  
🟡 **SHOULD**: スクリーンショットは Staging ビルドの実機またはシミュレーターで撮影し、Figma でテキストオーバーレイを追加する

## 6. バージョン管理とリリースノートの書き方

### バージョン番号の更新手順

バージョン更新は `scripts/bump-version.js` で行う。実装の全文は当該ファイルを参照。

```bash
# patch / minor / major を指定してインクリメント
node scripts/bump-version.js --type patch
# → version: 2.3.0 → 2.3.1
# → buildNumber: "230" → "231"
# → versionCode: 230 → 231
```

更新される値:
- `expo.version`（semver）
- `expo.ios.buildNumber`（単調増加の文字列）
- `expo.android.versionCode`（単調増加の整数）

🔴 **MUST**: 同じ `buildNumber` / `versionCode` で 2 回ビルドを提出しない（ストアがリジェクトする）  
🟡 **SHOULD**: バージョン更新は PR にして、ビルドナンバーのコンフリクトを防ぐ

### リリースノートのテンプレート

```
ユーザー向け（ストア公開用）:
  ─────────────────────────────
  【v2.3.0 アップデート】

  ■ 新機能
  ・注文履歴から再注文できるようになりました
  ・ダークモードに対応しました

  ■ 改善
  ・検索画面の表示速度を改善しました

  ■ 不具合修正
  ・特定の環境で決済画面が表示されない問題を修正しました
  ─────────────────────────────

社内向け（CHANGELOG.md / チケット番号）:
  - #1234 feat: 再注文機能を追加
  - #1289 feat: ダークモード対応（トークン更新）
  - #1301 perf: 検索 API のキャッシュ改善
  - #1312 fix:  決済 WebView の初期化タイミング修正
```

🟡 **SHOULD**: ユーザー向けノートはマーケティング担当がレビューしてからストアに入力する  
🔴 **MUST NOT**: 社内チケット番号・技術的な詳細をストアのリリースノートに書かない

## 7. 審査リジェクト対応フロー

### エスカレーションパス

```
リジェクト通知受信（App Store Connect メール / Play Console メール）
    ↓
担当エンジニアがリジェクト内容を分析（当日中）
    ↓
対応方針を判断
  ├─ 技術的修正で対応可能 → エンジニアが修正・再提出
  ├─ ガイドライン解釈が不明確 → TL に相談（翌営業日までに）
  ├─ IAP / プライバシーポリシーに関わる → 法務・PdM にエスカレーション
  └─ Apple に異議申し立てが必要 → TL + PdM が Resolution Center で対応

対応完了後: Slack #release チャネルに結果をアナウンスする
```

### リジェクト通知の受け取り方

```
iOS:     App Store Connect のメール通知 + Resolution Center
Android: Play Console のメール通知 + ダッシュボード
```

### リジェクト種別と対応方針

| リジェクト種別 | 典型的な原因 | 対応方針 |
|--------------|------------|---------|
| **Guideline 2.1** Performance（クラッシュ）| 特定デバイス・OS でのクラッシュ | Sentry / Crashlytics でエラーを特定して修正 |
| **Guideline 2.3** Accurate Metadata | スクリーンショットとアプリの内容が一致しない | スクリーンショットを更新 |
| **Guideline 4.0** Design | ネイティブ UI ガイドラインへの非準拠 | HIG を確認して修正 |
| **Guideline 5.1.1** Data Collection（プライバシー）| 権限説明文が不十分・プライバシーラベルの不整合・Privacy Manifest の申告漏れ | Section 3-1・3-2・3-3 を参照して修正 |
| **Guideline 3.1.1** In-App Purchase | デジタルコンテンツを外部決済 | IAP を実装、または Apple に該当しない旨を説明 |
| **Google Policy**: Metadata Policy | 説明文にキーワードの詰め込み | 自然な説明文に書き直す |
| **Google Policy**: Target API level | targetSdkVersion が要件を満たさない | Section 4-1 を参照して更新 |


## 8. TestFlight 運用（iOS）

### TestFlight の配布フロー

```
EAS Build（production プロファイル）
    ↓ EAS Submit
App Store Connect（ビルドのアップロード）
    ↓ 処理完了まで 15〜30 分
TestFlight（内部テスト）      ← 審査なし・即時配布
    ↓ グループ追加
TestFlight（外部テスト）      ← 審査なし・即時配布（2024 年以降 β 審査廃止）
    ↓ 確認完了後
App Store 審査提出
```

### 内部テスターとグループ管理

```
App Store Connect → TestFlight → 内部テスト
  └─ グループ: 開発チーム（最大 100 名）
       ├─ 自動配布: ON（ビルド提出後に自動通知）
       └─ メンバー: チームメンバーを Apple ID で招待

App Store Connect → TestFlight → 外部テスト
  └─ グループ: QA チーム
       └─ 最大 10,000 名（審査なし）
```

🟡 **SHOULD**: 内部テストグループに全開発者を追加し、毎 Staging ビルドで確認できるようにする

### TestFlight ビルドの有効期限

TestFlight ビルドは 90 日間有効。期限切れ前にリリースするか、新しいビルドを提出する。  
App Store Connect のダッシュボードで週次確認する。

## 9. ロールアウト監視と停止手順

### iOS の段階リリース（Phased Release）

```
App Store Connect → バージョン → バージョンリリース → 段階的リリースを使用する

スケジュール（自動）:
  Day 1〜2:  1%
  Day 3〜4:  2%
  Day 5〜6:  5%
  Day 7〜8:  10%
  Day 9〜10: 20%
  Day 11〜12: 50%
  Day 13〜14: 100%
  ※ 最大 7 日間で自動的に 100% になる（手動で一時停止・再開・即時公開が可能）
```

**段階リリースの停止・キャンセル**:

```
App Store Connect → バージョン → 段階的リリース
  ├─ 一時停止: 最大 30 日間停止可能（その後自動再開）
  ├─ 再開: 停止前の割合から継続
  └─ 即時公開: 全ユーザーに一括配布（停止中でも実施可能）
```

🟡 **SHOULD**: 新機能を含むリリースは段階リリースを使用し、クラッシュ急増時に即停止できる体制を整える

### Android の段階ロールアウト

```
確認する指標（Google Play Console → Android Vitals）:
  ├─ クラッシュ率: < 1.09%（Play の優良率基準）
  ├─ ANR 率:      < 0.47%（Play の優良率基準）
  └─ レーティング変動: 急激な低下に注意
```

**ロールアウト拡大の判断基準**:

```
1% から 10% へ拡大する条件（10% → 50% → 100% も同様）:
  □ クラッシュ率が前バージョン以下
  □ ANR 率が前バージョン以下
  □ Sentry / Crashlytics で新規エラーが発生していない
  □ レーティングが維持されている
  □ 前の割合から 24 時間以上経過している
```

**ロールアウトの停止・ロールバック**:

```
Google Play Console → リリース → 本番 → 現在のリリースを管理
  → 「ロールアウトを一時停止」または「ロールアウトを停止してロールバック」
```

**OTA ホットフィックスによる緊急対応**（ネイティブ変更を伴わない場合のみ）:

```bash
eas update --branch production \
  --message "Hotfix: クラッシュ修正" \
  --rollout-percentage 10
```

> **OTA 適用可能の条件**: JS バンドルおよびアセットの変更のみであること。  
> ネイティブコード（Swift / Kotlin / C++）・`app.json` のネイティブ設定・ネイティブモジュールの追加・削除は OTA 対象外。対象外の変更をストアビルドなしで配布すると動作不定になる。

🟡 **SHOULD**: クラッシュ率が前バージョン比 2 倍超になった場合は即座にロールアウトを停止し、Slack #release チャネルにアナウンスする

## 10. よくあるトラブルと対処

| 症状 | 原因 | 対処 |
|------|------|------|
| iOS 審査でクラッシュ指摘（Guideline 2.1）| 特定デバイスのみ再現するクラッシュ | Apple からのシンボルファイル（.ips）を Sentry で解析 |
| 「Missing Compliance」警告 | 暗号化輸出規制の申告が未設定 | Section 3-7 を参照 |
| Privacy Manifest のリジェクト | Required Reason API の申告漏れ | Section 3-2 を参照 |
| Android ビルドが `deobfuscation` エラー | ProGuard マッピングファイルが未アップロード | EAS の `androidAppBundle` 設定を確認 |
| App Store Connect に「Missing Screenshots」| 必須デバイスサイズのスクリーンショットがない | Apple 公式ドキュメントで現行要件を確認して追加（Section 5 参照）|
| 提出後に「Invalid Binary」エラー | ビルドの署名・バンドル ID が不一致 | `eas credentials` で証明書を再設定 |
| Play Console で「API レベルが低い」| targetSdkVersion が要件未達 | Section 4-1 を参照して更新 |
| TestFlight でビルドが表示されない | App Store Connect の処理待ち | 15〜30 分待ってからページを更新 |
| ロールアウト後にクラッシュ急増 | 本番特有の設定・データとの不整合 | ロールアウトを即停止 → OTA でホットフィックス（対象外なら再ビルド）→ 原因修正 |
