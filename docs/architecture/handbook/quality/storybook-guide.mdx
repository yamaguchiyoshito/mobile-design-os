## 2. ファイル配置ルール

### 配置場所の判断基準

```
コンポーネントの種類                      配置先
─────────────────────────────────────────────────────────
Design System（components/ui/）        コンポーネントと同階層
Feature コンポーネント（features/*/）   コンポーネントと同階層
画面全体（app/）                        __stories__/ ディレクトリ
共通レイアウト（components/templates/）コンポーネントと同階層
```

```
src/
  components/
    ui/
      Button/
        Button.tsx
        Button.stories.tsx      ← ✅ 同階層
        Button.test.tsx
        index.ts
      Card/
        Card.tsx
        Card.stories.tsx        ← ✅ 同階層

  features/
    order/
      components/
        OrderCard.tsx
        OrderCard.stories.tsx   ← ✅ 同階層
```

🔴 **MUST**: Story ファイルはコンポーネントと同じディレクトリに置く  
🔴 **MUST NOT**: `src/stories/` のような集中管理ディレクトリに置かない  
（コンポーネントと Story が離れると、リネーム時に片方が取り残される）

### Story の削除・リネーム時の手順

```
1. Story ファイルを削除 / ファイル名変更
2. pnpm storybook:build でビルドエラーがないことを確認
3. Chromatic の孤立スナップショットを手動削除（baselines が残存する）
   → Chromatic ダッシュボード > Snapshots > "Removed" フィルタ > Accept all
4. PR コメントに削除理由を記載する
```

## 4. Meta オブジェクトの書き方

```tsx
// src/components/ui/Button/Button.stories.tsx
import type { Meta, StoryObj } from '@storybook/react';
import { Button }              from './Button';

const meta: Meta<typeof Button> = {
  title:     'UI/Button',
  component: Button,

  // ── Figma 連携（Figma MCP で URL を取得して貼る）──────────────────
  parameters: {
    design: {
      type: 'figma',
      url:  'https://www.figma.com/design/XXXX/MyApp?node-id=1-23',
    },
  },

  // ── args のデフォルト値（コールバックは書かない → argTypes で定義する）──
  args: {
    label:   'ボタン',
    variant: 'primary',
    size:    'md',
    // onPress はここに書かない。argTypes の action: で定義する
  },

  // ── Controls パネルの設定 ─────────────────────────────────────────
  argTypes: {
    variant: {
      control:     'select',
      options:     ['primary', 'secondary', 'destructive'],
      description: 'ボタンの見た目のバリエーション',
    },
    size: {
      control:  'select',
      options:  ['sm', 'md', 'lg'],
    },
    disabled: {
      control: 'boolean',
    },
    loading: {
      control: 'boolean',
    },
    onPress: {
      action: 'pressed',  // Actions パネルに記録される。jest.fn() を直書きしない
    },
  },
};

export default meta;
type Story = StoryObj<typeof Button>;
```

### `parameters.design` の URL 取得方法（Figma MCP）

```bash
# Claude Code のターミナルで:
# 「Button コンポーネントの Figma URL を教えて」
# → Figma MCP が対応するノードの URL を返す
# → Story の parameters.design.url に貼り付ける
```

🔴 **MUST**: `onPress` 等のコールバックは `argTypes` の `action:` で定義する。`args` への `() => {}` 直書き・`jest.fn()` 直書きは禁止  
🔴 **MUST**: `parameters.design.url` は全 Design System コンポーネントに設定する  
🟡 **SHOULD**: `argTypes` の `description` は日本語で書く（Docs タブに表示される）

## 5. Story バリエーションの網羅方針

### 網羅すべき 5 軸

```
1. バリアント      → Figma の variant 定義を全カバー
2. サイズ          → sm / md / lg 等の全サイズ
3. 状態            → デフォルト / hover / active / disabled / loading / error
4. コンテンツ長    → 短いテキスト / 長いテキスト / 空 / 改行含む
5. データパターン  → 正常 / 空 / エラー / ローディング（API 連携の場合）
```

```tsx
// ── 1. バリアント ──────────────────────────────────────────────────
export const Primary: Story = {
  args: { variant: 'primary', label: '確認する' },
};
export const Secondary: Story = {
  args: { variant: 'secondary', label: 'キャンセル' },
};
export const Destructive: Story = {
  args: { variant: 'destructive', label: '削除する' },
};

// ── 2. サイズ ──────────────────────────────────────────────────────
export const Small: Story  = { args: { size: 'sm', label: '小' } };
export const Medium: Story = { args: { size: 'md', label: '中' } };
export const Large: Story  = { args: { size: 'lg', label: '大' } };

// ── 3. 状態 ───────────────────────────────────────────────────────
export const Disabled: Story = {
  args: { disabled: true, label: '送信できません' },
};
export const Loading: Story = {
  args: { loading: true, label: '送信中' },
};

// ── 4. コンテンツ長 ────────────────────────────────────────────────
export const LongLabel: Story = {
  args: { label: '非常に長いボタンラベルのテキストが入った場合の表示確認' },
};

// ── 5. アイコン付き ────────────────────────────────────────────────
export const WithIcon: Story = {
  args: {
    label:    'カートに追加',
    iconLeft: 'shopping-cart',
  },
};
```

### Feature コンポーネントのデータパターン

API データを持つ Feature コンポーネントは MSW で状態を制御する（Section 8 参照）。

```tsx
export const Default:  Story = { /* 正常データ */ };
export const Empty:    Story = { /* データが 0 件 */ };
export const Loading:  Story = { /* フェッチ中 */ };
export const Error:    Story = { /* API エラー */ };
```

🔴 **MUST**: Figma で定義された variant を Story で全カバーする  
🔴 **MUST**: `disabled` / `loading` / `error` の状態 Story を必ず作成する  
🟡 **SHOULD**: コンテンツ長バリエーションを Design System コンポーネントに追加する

## 7. Docs の書き方

### コンポーネント Docs（自動生成 + 手動補完）

JSDoc コメントは Docs タブに自動表示される。Design System コンポーネントの使い方・制約・NG パターンをここに集約することで、Figma や Confluence を参照しなくても Storybook 単体で仕様が分かる状態を目指す。

```tsx
/**
 * Button はユーザーのアクションを起動するための基本コンポーネントです。
 *
 * ## 使い分け
 * - `primary` — メインアクション（1 画面に 1 つまで）
 * - `secondary` — サブアクション・キャンセル
 * - `destructive` — 取り消しできない破壊的操作
 *
 * ## 注意事項
 * - `disabled` と `loading` を同時に設定しない
 * - ラベルは 20 文字以内を推奨
 */
const meta: Meta<typeof Button> = {
  title:     'UI/Button',
  component: Button,
};
```

## 8. MSW によるモック統合

Feature コンポーネントが API データを必要とする場合、MSW ハンドラを Story に組み込む。

### 基本パターン

```tsx
// src/features/order/components/OrderList.stories.tsx
import type { Meta, StoryObj }     from '@storybook/react';
import { http, HttpResponse, delay } from 'msw';
import { OrderList }               from './OrderList';
import { createOrderList }         from '@/mocks/factories/orderFactory';

// ベース URL が未設定の場合は空文字にフォールバック（MSW はパスマッチで動作する）
const BASE = process.env.EXPO_PUBLIC_API_BASE_URL ?? '';

const meta: Meta<typeof OrderList> = {
  title:     'Feature/Order/OrderList',
  component: OrderList,
};
export default meta;
type Story = StoryObj<typeof OrderList>;

// ── 正常系 ────────────────────────────────────────────────────────
export const Default: Story = {
  parameters: {
    msw: {
      handlers: [
        http.get(`${BASE}/orders`, () =>
          HttpResponse.json({
            items:      createOrderList(5),
            nextCursor: null,
            hasMore:    false,
          }),
        ),
      ],
    },
  },
};

// ── 空の状態 ─────────────────────────────────────────────────────
export const Empty: Story = {
  parameters: {
    msw: {
      handlers: [
        http.get(`${BASE}/orders`, () =>
          HttpResponse.json({ items: [], nextCursor: null, hasMore: false }),
        ),
      ],
    },
  },
};

// ── ローディング（遅延レスポンス）────────────────────────────────
export const Loading: Story = {
  parameters: {
    msw: {
      handlers: [
        http.get(`${BASE}/orders`, async () => {
          await delay('infinite');    // 永久にローディング状態を維持
          return HttpResponse.json({});
        }),
      ],
    },
  },
};

// ── API エラー ────────────────────────────────────────────────────
export const NetworkError: Story = {
  parameters: {
    msw: {
      handlers: [
        http.get(`${BASE}/orders`, () => HttpResponse.error()),
      ],
    },
  },
};

export const ServerError: Story = {
  parameters: {
    msw: {
      handlers: [
        http.get(`${BASE}/orders`, () =>
          HttpResponse.json({ message: 'Internal Server Error' }, { status: 500 }),
        ),
      ],
    },
  },
};
```

### QueryClientProvider のデコレーター

```tsx
// .storybook/preview.ts（web 系）
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { initialize, mswLoader }            from 'msw-storybook-addon';

initialize();

export const decorators = [
  // ⚠️ QueryClient は Story ごとに新規生成する。
  //    グローバルで 1 インスタンスを使い回すと Story 間でキャッシュが汚染される。
  (Story) => {
    const queryClient = new QueryClient({
      defaultOptions: { queries: { retry: false, staleTime: 0 } },
    });
    return (
      <QueryClientProvider client={queryClient}>
        <Story />
      </QueryClientProvider>
    );
  },
];

export const loaders = [mswLoader];
```

🔴 **MUST**: API 依存の Feature Story には正常系・空・ローディング・エラーの 4 パターンを用意する  
🔴 **MUST**: Story に MSW ハンドラを書く場合は `parameters.msw.handlers` に定義する（グローバルハンドラを Story 内で上書きしない）  
🔴 **MUST**: `QueryClient` は Story ごとに新規生成する。グローバルシングルトンにしない  
🟡 **SHOULD**: `delay('infinite')` でローディング状態を再現し、Skeleton の表示を確認する

import { useEffect }       from 'react';   // ← 必ずインポートする
import { I18nextProvider } from 'react-i18next';
import i18n                from '@/lib/i18n';

export const globalTypes = {
  locale: {
    description: '表示言語',
    defaultValue: 'ja',

      items: [
        { value: 'ja', title: '日本語' },

  (Story, context) => {

      <I18nextProvider i18n={i18n}>

      </I18nextProvider>
    );
  },
];

## 10. アクセシビリティの確認手順

### addon-a11y による自動チェック（web Storybook）

```ts
// .storybook/preview.ts — a11y のグローバル設定
export const parameters = {
  a11y: {
    config: {
      rules: [
        { id: 'color-contrast', enabled: true },
        { id: 'button-name',    enabled: true },
        { id: 'image-alt',      enabled: true },
      ],
    },
  },
};
```

```tsx
// Story レベルの a11y 設定（特定ルールを無効化する場合のみ）
export const DisabledState: Story = {
  args: { disabled: true, label: '送信できません' },
  parameters: {
    a11y: {
      config: {
        rules: [
          // disabled 状態はコントラスト比が低くても意図的なため除外
          { id: 'color-contrast', enabled: false },
        ],
      },
    },
  },
};
```

### a11y チェックの手順

```
1. pnpm storybook:web で web Storybook を起動する
2. 確認したい Story を開く
3. 下部の「Accessibility」タブを開く
4. 「Violations」が 0 件になることを確認する
5. PR マージ前に全 Story で 0 件を維持する（CI で自動チェック済み）
```

### Story に含めるべき a11y 確認項目

```tsx
// ✅ accessibilityRole の確認が取れる Story を必ず作る
export const AsButton: Story = {
  args: { label: '送信する' },
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement);
    expect(canvas.getByRole('button', { name: '送信する' })).toBeInTheDocument();
  },
};

// ✅ disabled 状態の accessibilityState 確認
export const DisabledA11y: Story = {
  args: { label: '送信する', disabled: true },
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement);
    const button = canvas.getByRole('button');
    expect(button).toHaveAttribute('aria-disabled', 'true');
  },
};
```

🔴 **MUST**: 全 Story で addon-a11y の Violations が 0 件であること  
🟡 **SHOULD**: インタラクティブコンポーネントの Story に Play 関数で `getByRole` 確認を追加する

## 11. Play 関数（インタラクションテスト）

### 書くべき場面

### テンプレート

```tsx
import { expect, userEvent, within } from '@storybook/test';

export const ShowsValidationError: Story = {
  args: {
    label:        'メールアドレス',
    value:        '',
    onChangeText: () => {},
  },
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement);
    const input  = canvas.getByLabelText('メールアドレス');

    await userEvent.type(input, 'invalid-email');
    await userEvent.tab();   // blur でバリデーション発火

    await expect(canvas.getByRole('alert')).toBeVisible();
    await expect(canvas.getByText('正しいメールアドレスを入力してください')).toBeVisible();
  },
};

export const ClearsErrorOnValidInput: Story = {
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement);
    const input  = canvas.getByLabelText('メールアドレス');

    await userEvent.type(input, 'valid@example.com');
    await userEvent.tab();

    await expect(canvas.queryByRole('alert')).not.toBeInTheDocument();
  },
};
```

🟡 **SHOULD**: フォームコンポーネントの全 Story に Play 関数を追加する  
🟡 **SHOULD**: Play 関数は `userEvent` を使う。`fireEvent` は実際のユーザー操作と乖離するため使わない

## 12. VRT（Visual Regression Testing）

Chromatic を使って Story のスクリーンショットを自動比較する。スタイル変更の意図しない影響を検出することが目的で、Play 関数の「動作確認」とは役割が異なる。

### 実行タイミング

| タイミング | コマンド / 方法 |
|---|---|
| PR 時（自動） | GitHub Actions から Chromatic CI が実行される |
| ローカル確認 | `npx chromatic --project-token=<TOKEN>` |

### VRT の承認フロー

```
1. PR を作成すると Chromatic CI が自動実行される
2. 差分が検出された Story は Chromatic ダッシュボードに表示される
3. 意図した変更 → レビュアーが「Accept」する
4. 意図しない変更 → コードを修正して再 Push する
5. 全差分が Accept されると CI がグリーンになる
```

### Story で VRT を無効化する場合

```tsx
// 特定 Story を VRT 対象外にする（アニメーション・ランダムデータを含む Story 等）
export const AnimatingState: Story = {
  parameters: {
    chromatic: { disableSnapshot: true },
  },
};
```

🔴 **MUST**: Chromatic の未承認差分がある状態で PR をマージしない  
🟡 **SHOULD**: アニメーション・乱数・日時依存の Story は `disableSnapshot: true` を設定する

## 13. on-device Storybook への追加手順

web Storybook と同じ `.stories.tsx` ファイルを読み込むため、基本的に追加作業は不要。  
ただし on-device 固有の設定が必要な場合がある。

### on-device でのデコレーター

```tsx
// .rnstorybook/preview.ts（on-device 系）
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { I18nextProvider }                  from 'react-i18next';
import i18n                                 from '@/lib/i18n';

export const decorators = [
  (Story) => {
    // web 系と同様、Story ごとに新規生成する
    const queryClient = new QueryClient({
      defaultOptions: { queries: { retry: false } },
    });
    return (
      <QueryClientProvider client={queryClient}>
        <I18nextProvider i18n={i18n}>
          <Story />
        </I18nextProvider>
      </QueryClientProvider>
    );
  },
];
```

### Platform 分岐が必要な場合の方針

on-device と web で挙動が異なる場合は、**Story 内でプラットフォーム分岐を書かない**。  
差異はコンポーネント実装側（`Platform.select` 等）に閉じ込め、Story からは見えないようにする。  
どうしても Story レベルの分岐が必要な場合は、その理由をコメントで記載して例外申請する。

```tsx
// ❌ 禁止 — Story 内でプラットフォーム分岐を書くと Story が環境依存になる
export const Default: Story = {
  args: {
    icon: Platform.OS === 'web' ? undefined : 'checkmark',
  },
};

// ✅ 正解 — コンポーネント側で吸収する
// MyIcon.tsx:
//   const icon = Platform.select({ native: 'checkmark', default: undefined });
// Story:
export const Default: Story = {
  args: {},  // プラットフォーム差異はコンポーネント側が吸収している
};
```

🟡 **SHOULD**: on-device で確認すべき項目（Dynamic Type・ジェスチャー・フォント）は  
Story に `// on-device 確認必須: <理由>` コメントを残す

## 14. Story 追加のステップバイステップ

### 手動で追加する場合

```
Step 1: Figma で仕様確認（Figma MCP）
  └─ Claude Code: 「この Figma URL のコンポーネント仕様を教えて」
  └─ variant・size・状態・props を把握する

Step 2: Meta オブジェクトを書く
  └─ title（カテゴリ/コンポーネント名）
  └─ argTypes（Controls 設定、コールバックは action: で定義）
  └─ parameters.design.url（Figma URL）

Step 3: 必須 Story を追加する（最低限）
  └─ デフォルト状態
  └─ 全 variant
  └─ disabled / loading / error 状態

Step 4: 追加 Story を追加する（推奨）
  └─ コンテンツ長バリエーション
  └─ アイコン付き / なし
  └─ API データパターン（Feature コンポーネントのみ・Section 8 参照）

Step 5: Play 関数を追加する（インタラクションがある場合）
  └─ ユーザー操作フロー
  └─ バリデーションエラーの表示・消去

Step 6: web Storybook で確認する（pnpm storybook:web）
  └─ Controls タブで全バリアントを手動確認
  └─ Accessibility タブで Violations = 0 を確認
  └─ Docs タブで自動生成された説明を確認・補足

Step 7: on-device Storybook で確認する（Design System コンポーネントのみ必須）
  └─ EXPO_PUBLIC_STORYBOOK_ENABLED=true pnpm start
  └─ Dynamic Type 最大サイズで崩れないことを確認

Step 8: CI で確認する（PR 時自動実行）
  └─ addon-vitest（Play 関数テスト）
  └─ addon-a11y（Violations チェック）
  └─ Chromatic（VRT 差分確認・承認）
```

## 15. Claude Code + MCP を使った Story 自動生成

### Storybook MCP + Figma MCP の組み合わせ

自動生成を使う場合、Section 14 の Step 1（Figma 仕様確認）・Step 2（Meta 作成）・Step 3（必須 Story 追加）を一括で代替できる。  
Step 5 以降（Play 関数・web 確認・CI）は自動生成後も手動で行う。

```bash
# Claude Code のターミナルで以下を入力:

「次の条件で Button コンポーネントの Story ファイルを生成して:
  - Figma URL: https://www.figma.com/design/XXXX?node-id=1-23
  - 既存の Card コンポーネントの Story（src/components/ui/Card/Card.stories.tsx）
    を命名規則の参考にする
  - MSW モックは不要
  - Play 関数は onPress のテストを含める」
```

### 生成後のチェックポイント

```
□ title が命名規則に従っている（例: UI/Button）
□ argTypes が全 props を網羅している
□ コールバックが argTypes の action: で定義されている（args への直書きがない）
□ Figma URL が parameters.design.url に設定されている
□ disabled / loading / error の Story がある
□ Play 関数が getByRole を使っている（getByTestId ではない）
□ QueryClient がデコレーター内で毎回 new されている
```
